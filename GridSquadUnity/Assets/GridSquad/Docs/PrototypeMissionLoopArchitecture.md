# 임무 루프 아키텍처

## 제품 규칙

- 기지에서 1~3명을 선발하고 하나의 임무에 출발한다.
- 임무는 순서가 고정된 여러 스테이지로 구성된다.
- 체력, 탄약, 소모품, 장비 내구도는 스테이지 사이에 유지된다.
- 임무가 끝나 기지로 복귀하면 체력만 완전히 회복된다.
- 실제 체력 피해는 `피해량 × 트라우마 배율 + 고정 트라우마`로 누적된다.
- 트라우마 저항은 최대 체력 기준의 후유증 단계 임계값을 높인다.
- 체력이 0이 된 대원은 임무 중 전투 불능이며 영구 사망하지 않는다.
- 전투 불능 대원은 최소 심각 단계 후유증을 받는다.
- 한 임무에서는 가장 높은 단계의 후유증 하나만 판정한다.
- 같은 후유증이 다시 나오면 더 긴 휴식 횟수를 유지하고, 다른 후유증은 함께 존재한다.
- 후유증 휴식 횟수는 해당 대원이 출격하지 않은 임무가 끝날 때만 감소한다.
- 내구도 0 장비는 장착 상태와 저장 데이터는 유지하지만 기능과 능력치 보너스를 제공하지 않는다.
- 장비 수리는 기지에서만 수행한다. 현재 프로토타입의 수리 비용은 무료다.

## 순차 초기화

`PrototypeGameApplication`이 유일한 상위 흐름 소유자다.

1. 콘텐츠 카탈로그의 ID 인덱스를 검증한다.
2. 기지 영속 상태를 생성한다.
3. 선발 결과로 임무 스냅샷을 만든다.
4. 스테이지 씬을 로드한다.
5. 미사용 전투원을 그리드에서 등록 해제한다.
6. 배치 전투원에 체력·아이템·탄약·내구도 상태를 주입한다.
7. `MissionCombatantStateBridge`를 연결한다.
8. `CombatDirector`에 이번 스테이지 전투원만 등록한다.
9. 모든 준비가 끝난 뒤 행동 그래프와 전투를 시작한다.
10. 전투 종료 시 체력과 아이템 상태를 임무 스냅샷에 기록한다.
11. 다음 스테이지, 철수, 패배 또는 임무 완료를 판정한다.
12. 임무 정산 후 기지 씬으로 복귀한다.

`CombatDirector`의 독립 실행 호환성은 유지한다. 영속 애플리케이션이 없는 상태로 전투 씬만 실행하면 기존 방식으로 한 프레임 뒤 전투를 시작한다.

## 데이터 경계

- 정적 콘텐츠: `GameContentCatalog`, `MissionDefinition`, `EncounterDefinition`, `AftereffectRuleSet`
- 기지 영속 상태: `BaseState`, `UnitState`, `ItemState`, `EquipmentSlotState`, `AftereffectState`
- 임무 영속 상태: `ActiveMissionState`, `MissionUnitState`
- 세션 루트: `GameSessionState`

영속 상태는 씬 오브젝트나 `ScriptableObject` 참조를 저장하지 않는다. 문자열 ID, 숫자, 불리언, 리스트만 보유하므로 이후 JSON 또는 바이너리 저장 계층을 붙일 때 씬과 콘텐츠 에셋을 직렬화할 필요가 없다. `dataVersion`은 마이그레이션 진입점이다.

## 확장 지점

- 저장/불러오기: `GameSessionState`를 대상으로 버전별 DTO 변환기와 원자적 파일 교체를 추가한다.
- 유료 수리: `EquipmentRepairService` 앞에 자원 비용 정책을 주입한다.
- 후유증 추가: 정의 에셋만 추가하고 규칙 세트에 등록한다.
- 임무 추가: 조우와 스테이지 정의를 조합해 임무 에셋을 추가한다.
- 스테이지별 다른 맵: `MissionStageDefinition.scenePath`만 다른 씬 경로로 지정한다.
